Part 2 · 怎么做

Part 2  ·  架构设计

系统分为两层：Layer 1 评论问答引擎（负责检索和生成） + Layer 2 场景顾问 Agent（负责理解和决策）。 两层通过一个清晰的 query → answer 接口通信，各自独立演进。

Layer 2场景顾问 · 设计思路
核心问题：用户真正关心什么？

同一家酒店，出差客关心网络和退房速度，蜜月客关心景观和私密性。 Layer 2 做的事：
用户输入→ 场景识别→ 查矩阵选维度→ 生成 N 个子问题→ 并行发给 Layer 1→ 综合研判→ 结构化回复

Layer 1问答引擎 · 设计思路
核心问题：给定一个问题，怎么找到最相关的评论并生成回答？

Layer 1 是无状态服务——不关心谁在问、为什么问，只管检索和生成：
检索管线
query→ 五路检索→ RRF融合→ 重排序→ LLM生成→ answer
五路并行检索，各自互补：
BM25jieba分词 + 倒排索引，关键词精准匹配
向量语义text-embedding-v4 + ChromaDB，捕获同义表达
反向Query用 LLM 预生成的问题索引评论，「搜问题找答案」
HyDE生成假设评论再检索，弥补短查询 vs 长文档的语义鸿沟
类别摘要14类预生成摘要，面对概述型 query 提供全局视野

交互逻辑：Layer 2 如何调用 Layer 1

场景评估时，Layer 2 并行发起多个子查询
Layer 2 识别场景为「商务出差」→ 选出网络/安静/退房/位置四个维度 → 生成四个自然语言 query
↓ 并发调用 Layer 1 ↓
📶 「WiFi稳不稳？」
🔇 「隔音好不好？」
🏃 「退房快不快？」
📍 「交通方便吗？」
↓ 返回四个带引用的回答 ↓
Layer 2 综合四个维度的结果，研判总体是否推荐，按模板生成结构化回复
接口契约
# Layer 1 对外接口
def ask(
  query: str,           # "朝花园安静吗？"
  filters: dict | None,# 可选: 房型/时间过滤
  strategy: str = "hybrid"
) → dict:
  # { answer: str, sources: list, metrics: dict }

# Layer 2 对外接口
def advise(
  user_input: str,   # "下周出差三天…"
  session_id: str
) → dict:
  # { scene, dimensions[], verdict, reply }
关键：Layer 1 的接口是纯文本问答，不感知场景。
场景逻辑全部在 Layer 2 实现。
加新场景不改引擎，换检索引擎不改顾问。
