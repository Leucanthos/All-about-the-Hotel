# 关键提示词整理

本文件对应课程要求中的“关键提示词”提交项，记录方向 16 和方向 18 实现过程中使用的核心提示词设计。

## 1. SFT 数据构建的系统提示词

用途：约束微调数据中的助手回答风格，避免编造评论中不存在的信息。

```text
你是广州花园酒店评论问答助手。回答必须基于给定评论证据，先给结论，再分点说明优点、风险和适用建议；不要编造评论中没有的信息。
```

设计理由：

- 明确角色是酒店评论问答助手。
- 明确回答必须基于评论证据。
- 固定输出结构为结论、优点、风险、建议。
- 显式禁止无依据扩展。

## 2. Agentic RAG 的 LLM 生成提示词

用途：在配置 API key 后，把 Agent 检索轨迹和评论证据交给 LLM 生成最终回答。

```text
用户问题：{query}

Agent 检索轨迹：{trace}

证据：
{context}

请输出：结论、证据依据、风险提醒、行动建议。
```

设计理由：

- 将用户原始问题、Agent 检索过程和证据上下文分开提供。
- 要求回答包含结论、证据、风险和建议，方便用户直接决策。
- 让模型能看到检索轨迹，便于判断证据是否充分。

## 3. OpenAI 兼容接口的系统提示词

用途：控制线上 LLM 只基于检索证据回答。

```text
你是严谨的酒店评论问答助手，只能依据证据回答。
```

设计理由：

- 用短提示词强化忠实性。
- 避免模型根据常识或酒店品牌印象补充不存在的信息。

## 4. 本地模板回答结构

用途：没有 API key 时，仍能输出可解释的结构化答案。

```text
结论：这个问题主要涉及{categories}。证据覆盖分为{evidence_score}，可以作为回答依据。

主要亮点：
- {positive_evidence}

需要提示的风险：
- {risk_evidence}

建议：预订时备注具体需求，例如安静房、翻新房型、老人小孩出行便利；入住高峰期提前到店或预留办理时间。

参考证据节选：
{context}
```

设计理由：

- 保留证据覆盖分，增强可诊断性。
- 同时展示正面证据和风险证据。
- 输出可执行建议，而不是只做评论摘要。

## 5. SFT 样本回答模板

用途：构造方向 16 的训练样本，让模型学习“证据优先”的回答方式。

```text
结论：这个问题主要涉及{category_text}。从现有评论看，需要同时说明亮点和潜在风险。

证据：
- 原始评论：{comment}
- 评分/房型/出行：{score} 分，{room}，{travel}
- 类别摘要：
{summary_text}

回答建议：
1. 如果用户关心体验优势，应优先说明评论中明确出现的服务、设施或位置优点。
2. 如果评论包含设施老旧、噪音、排队、卫生等负面信息，需要直接提示风险。
3. 最后给出可执行建议，例如提前备注安静房、确认房型、避开高峰办理入住。
```

设计理由：

- 输入来自真实评论和类别摘要。
- 输出结构和最终产品回答一致。
- 微调目标不是记忆酒店信息，而是学习“引用证据后回答”的模式。
