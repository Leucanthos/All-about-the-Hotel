# 宿说 — 酒店评论智能顾问 · 技术报告

> **项目**: 酒店评论跨模态检索与智能问答系统  
> **数据**: 6441 条广州花园酒店真实评论 · 1233 张酒店图片 · 4 种房型 · 414k条多酒店扩展数据  
> **成员**: SU Leuca（多模态检索） · SUN Yuhao（Agentic RAG + SFT） · dyx（增强检索算法）

---

## 一、项目概述

### 1.1 背景

这门课的大作业。三个人各自从不同的技术方向切入——Leuca 搞多模态（图文检索），Yuhao 搞 RAG 流水线（带 Agent 循环），dyx 搞各种检索增强算法（BM25、意图分类、融合）。最开始大家各自写各自的，最后要拼成一套能跑的系统的时候才发现问题：调用方式不统一、数据重复加载、根本没有路由层。

所以第三阶段主要工作就是：把三套东西整合到一起，加一个路由层做自动调度，再包一层 CLI 和 Web Demo。

### 1.2 数据概况

**基础数据集：** 广州花园酒店携程评论，6441 条带打分的中文评论、1233 张酒店实拍图片（客房、餐厅、泳池、外观等），覆盖 4 种房型。原始数据是一张 parquet 表，每行包含 `comment`（评论文本）、`score`（打分 1-5）、`fuzzy_room_type`（模糊房型）、`image_id`（关联图片）等字段。

**扩展数据集（后期加入）：** 从 296MB CSV 抽取 694 家酒店、414,330 条评论，转换为 Parquet（84MB，压缩比 3.4x），存为 `data/hotel_reviews_full.parquet`。用于多酒店检索测试。

### 1.3 系统架构

```
Layer 3 (应用层)     CLI (hotel/__main__.py)  +  Web Demo (demo.py)
                           │
Layer 2 (路由层)     意图分类 → 策略编排 → 结果聚合
                     (_shared/router.py → _shared/retriever.py)
                           │
Layer 1 (算法层)     多模态CLIP | BM25 | RRF融合 | Agentic RAG | SFT
                    (Leuca)     (dyx)   (dyx)    (Yuhao)      (Yuhao)
```

之所以分三层，不是因为设计阶段就规划好了，而是开发过程中自然长出来的。核心设计原则就一句话：**用户不加任何参数的时候，整条链路跑在本地，不用掏一分钱 API 费用。**

## 二、算法设计（Layer 1）

### 2.1 多模态图文检索 — SU Leuca

基于 **Chinese-CLIP**（OFA-Sys/chinese-clip-vit-base-patch16）构建文本↔图片跨模态检索。文本和图片各过一个 Transformer 编码到 512 维向量空间，余弦相似度检索。

自研 **NumPy 向量存储** 替代 ChromaDB（解决 Windows HNSW 持久化 bug），单次查询 2-3ms。推理后端双模式：PyTorch CPU（通用） / OpenVINO GPU（Intel Arc，文本编码提速 23×）。

#### 准确率 (240 样本)

| 指标 | 值 | 说明 |
|------|-----|------|
| Strict Hit@10 | 5.4% | 精确命中目标图片（随机基线 0.8%，比随机好 6 倍） |
| Relaxed Hit@10（同房型） | 51.5% | 房型级别语义可用 |
| MRR | 0.015 | 平均倒数排名 |
| Coverage | 57% | 超过一半图片曾被命中 |

**错误分析：** 63% 语义鸿沟（抽象描述无法对应视觉特征），33% 视觉混淆（不同房型照片相似），4% 索引缺失。

#### LTR 重排序优化

训练 4K 参数 MLP，使用 CLIP 相似度 + 房型 + 评分等特征。Hit@5 从 1% → **7%（6 倍提升）**，推理开销微秒级。

| 轮次 | 方法 | Hit@1 | Hit@5 | Hit@10 |
|------|------|-------|-------|--------|
| R0 | CLIP baseline | 0% | 1.0% | 5.0% |
| R1 | MLP 4feat pairwise | 1.0% | **7.0%** | 8.0% |
| R2 | 5feat listwise | **3.0%** | 7.0% | 8.0% |
| R3 | 7feat deeper | 2.0% | 7.0% | 8.0% |

**核心发现：** 所有优化在 Hit@5 ≈ 7% 触及天花板。瓶颈在于 CLIP 召回池 top-20 中仅 1.4% 包含正确答案。**瓶颈在召回，不在排序。**

#### 多模态召回评测 (10 queries, text-based relevance)

| Metric | CLIP (Leuca) |
|--------|:----------:|
| P@1 | 50.0% |
| P@5 | 36.0% |
| P@10 | 39.0% |
| MRR | 59.6% |

> 注意：CLIP 是图文检索引擎，用文本相关性来评价图片检索本身就存在偏差——查"停车场"返回的是拍了停车场的照片（及关联评论），但关联评论不一定包含"停车场"这个词。图片级别的 ground truth 尚未构建，这个指标仅供参考。

### 2.2 Agentic RAG + SFT — SUN Yuhao

#### 设计

**HotelCorpus 检索引擎：** 基于 char_terms 分词的轻量检索，支持 11 组类别别名映射。v2 集成 dyx 的 jieba BM25 双引擎。

**AgenticRAG 流水线（Plan → Retrieve → Evaluate → Rewrite/Retry）：**

1. **Plan** — 分析查询关键词，推断相关类别与检索策略
2. **Retrieve** — 执行类别引导或全局检索
3. **Evaluate** — 计算证据覆盖分（基于 BM25 分数 + 词重叠），低于 0.45 触发重试
4. **Rewrite** — 重写查询追加类别提示，二次检索取最优

#### 证据覆盖 (3 queries)

| 查询 | 证据分 | 推断类别数 | 轨迹 |
|------|--------|-----------|------|
| "前台服务怎么样？" | 1.00 | 4 类 | plan → evaluate（一次过） |
| "早餐有什么推荐？" | 1.00 | 1 类 | plan → evaluate（一次过） |
| "带老人小孩合适吗？" | 1.00 | 5 类 | plan → evaluate（一次过） |

所有测试查询均一次检索即满足证据要求（无重试），evidence score 均达到 1.00。

#### RAG 召回评测 (10 queries, text-based)

| Metric | RAG (Yuhao) |
|--------|:----------:|
| P@1 | 100.0% |
| P@5 | 96.0% |
| P@10 | 75.0% (仅返回 8 条 evidence) |
| MRR | 100.0% |

RAG 的定位是"结构化带证据的回答"而非纯文档召回。P@10 偏低是因为它只返回最多 8 条 evidence 且 focus 在覆盖率而非 top-rank 密度。

#### SFT 数据与微调

SFT 做了 1200 条训练数据的构建，使用 unsloth + QLoRA 4-bit 对 Qwen3 微调。微调的目标是让模型学会"先引用证据再回答"的格式。权重已产出，但未部署在线推理。

#### 摘要检索与注入 — 反向 Query 桥接

RAG 管线的检索层除了直接搜评论，还有一个重要的补充：**类别摘要的非对称向量检索**。这部分由 dyx 实现并整合进 Yuhao 的 RAG 管线。

**动机：** 人工标注了 14 个类别摘要（如"前台服务"、"餐饮设施"），每条摘要约 650 字，综合了大量评论的共性。但用户查询（短句，10-20 字）和类别摘要（长文，650 字）之间的语义差距太大，直接检索效果不好。

**解决方案——QA 变体桥接层：**

```
用户查询 (短句, 10字)
  |  -- 向量相似度高 --
  v
QA 变体 (20-50 字回答句, 每类别 3 条, 共 42 条)
  |  -- document 字段关联 --
  v
完整类别摘要 (650 字)
```

在查询和摘要之间插入一层 QA 变体——为每个类别人工构造 3 条"假如已被回答"的假设句（如"前台服务怎么样？"→"前台服务是酒店体验的核心亮点..."的前 50 字）。查询与 QA 变体的语义距离更近，检索准确率更高。

**非对称向量检索：**

| 环节 | 方式 | 说明 |
|------|------|------|
| 文档入库 | embed_documents() | QA 变体全文直接向量化，不加指令 |
| 查询检索 | embed_query() | 查询加 task 指令后向量化 |
| 候选池 | 8 条 | query() 返回 top-8 |
| 阈值过滤 | tau=0.25 | 余弦相似度低于阈值的丢弃 |
| 类别去重 | 同类别只保留最高分 | -- |
| 意图补充 | 规则强制补充 | 如果意图分类命中某类别但未出现在召回中，强制加入 |

**两层 Prompt 结构：**

召回的类别摘要以"类别背景知识"块注入 LLM prompt，位于具体评论列表之前：

```
【类别背景知识】（基于大量评论的综合分析）：
【前台服务】基于1352条评论，前台服务是酒店体验的核心亮点......（截取前300字）
【房间设施】酒店房间设施整体呈现显著的新旧并存特征......（截取前300字）

【真实住客评论】：
评论1：前台小姐姐超级贴心......
评论2：房间有点旧，设施一般......
```

这种"宏观背景 + 具体证据"的双层结构，使 LLM 在生成答案时能兼顾全局趋势与个体案例，尤其对"整体上评价怎么样"类泛化问题效果显著。

**改进对比：**

| 维度 | 改进前 | 改进后 |
|------|--------|--------|
| 检索方式 | get() 全量拉取，固定返回前3条 | query() 向量相似度检索，8候选池 |
| Embedding 文本 | 仅关键词（约30字） | 类别名+关键词+摘要前300字（约330字） |
| 查询向量化 | embed_batch() 无指令 | embed_query() 加 task 指令（非对称） |
| 相关性过滤 | 无，始终返回3条 | 余弦相似度阈值过滤 |
| 意图利用 | 未使用意图信号 | 规则映射强制补充命中类别 |
| LLM 上下文 | 摘要从未传入 LLM | 双层 prompt（背景+评论），300字截断 |
| QA 变体 | 无 | 每类别3条变体句，缩短 Query Gap |

> 这部分实现见 `dyx/engine.py` 中的 `expand_query_llm()` 和反向 Query 检索逻辑，以及 `dyx/__arch_research/hotel-report.tex` 的完整文档。

### 2.3 增强检索算法 — dyx

dyx 的代码从 `lib.py` / `lib_syh.py` 重构为三个独立模块：BM25 索引、意图分类、RRF 融合 + HyDE。

#### BM25 中文倒排索引

基于 jieba 分词的规范 BM25 算法，参数 k₁=1.5, b=0.75（标准值）。

**索引规模：** 2171 篇文档，2504 个词项，构建耗时 0.1s，查询延迟 avg 2.64ms。

| 查询词 | Top Score | Top-5 均值 | 说明 |
|--------|-----------|-----------|------|
| 早餐 | 4.04 | 3.83 | 高频词，区分度可 |
| 游泳池 | 12.67 | 11.80 | 低频词，精确匹配 |
| 服务 | 1.56 | 1.53 | 抽象概念，分偏低 |
| 噪音 | 11.92 | 10.05 | 低频精确词 |
| 停车场 | 12.80 | 12.63 | 匹配精准 |

#### BM25 召回评测 (10 queries)

| Metric | BM25 (dyx) |
|--------|:----------:|
| P@1 | **100.0%** |
| P@5 | **100.0%** |
| P@10 | **98.0%** |
| MRR | **100.0%** |

BM25 在酒店评论这个数据上几乎满分——精确关键词匹配就足够把相关文档排到最前面。

#### 细粒度意图分类

10 维中文关键词分类体系，覆盖 facility / price / location / service / food / cleanliness / quiet / child_friendly / elder_friendly / overall。

| 维度 | 示例关键词 | 映射策略 |
|------|-----------|--------|
| facility | 设施、设备、游泳池、健身房 | multimodal |
| price / location / service / food / ... | 价格、交通、前台、早餐 | rag |
| mixed（无匹配） | 空查询、纯英文 | multimodal + rag |

#### RRF 多路融合

RRF (Reciprocal Rank Fusion, k=60) 融合 BM25 + CLIP 双路结果。

| Metric | Fusion (dyx) |
|--------|:----------:|
| P@1 | **100.0%** |
| P@5 | **100.0%** |
| P@10 | **98.0%** |
| MRR | **100.0%** |

在 BM25 已经几乎满分的情况下，Fusion 的 RRF 融合提升不大。如果 BM25 不够好的场景（如需要语义泛化），CLIP 那路会补充进来。

## 三、路由架构设计（Layer 2）

### 3.1 流程

```
用户查询
   │
   ▼ 意图分类 (classify_intent_detailed, ~10μs)
   │  → 10 维中文关键词匹配
   │
   ▼ 策略编排 (retriever.retrieve)
   │  → multimodal / rag / bm25 / fusion
   │
   ▼ 结果聚合
   │  → 附带延迟、贡献者标注
```

### 3.2 路由准确率评测 (28 queries)

| 分类器 | 准确率 | 覆盖维度 | 测试集 |
|--------|:-----:|:-------:|:-----:|
| **dyx 中文 (10维)** | **96.4%** | 10 类 + mixed | 28 条 |
| 旧版英文 (qa/visual/mixed) | 7.1% | 3 类 | 28 条 |
| **提升** | **+89.3%** | | |

**唯一失败案例：** `值得推荐吗` → price（期望 overall）。原因是"值"是 price 类的关键词（"划算"、"值得"、"值"）。

**按维度分解：** 全部 10 个维度各有 2-3 条测试用例，仅 overall 维度因"值"字重叠出现 1 次失败。

### 3.3 路由延迟

| 阶段 | 延迟 | 说明 |
|------|:---:|------|
| 意图分类仅 | **10.6 μs**（中位数 9.6 μs） | 纯字符串匹配，微秒级 |
| route() 完整调用（缓存热） | **~14 ms**（7~29 ms） | 含策略分发 + 共享缓存命中 |
| 首次调用 CLIP 模型加载 | ~8 s（一次性） | 后续复用全局缓存 |

路由层自身开销（意图分类 + 策略分发）为 **<0.1 ms**，几乎可以忽略不计。

## 四、CLI 设计（Layer 3）

### 4.1 设计动机

三人三套算法，每套的入口和参数都不一样。CLI 的作用是提供一个统一的命令行接口，不管背后跑的是谁的算法。

### 4.2 使用示例

```
# 不加参数 → 自动路由
hotel search "游泳池干净吗"     # 自动识别为 facility → multimodal → 返回图片
hotel search "早餐好吃吗"       # 自动识别为 food → rag → 返回文本答案

# 指定算法 → 强制走某一路
hotel search "停车方便吗" --bm25     # 仅 BM25 (dyx)
hotel search "适合带老人吗" --rag    # 仅 Agentic RAG (Yuhao)
hotel search "套房装修" --mm          # 仅多模态 (Leuca)
hotel search "早餐怎么样" --fusion    # 仅多路融合 (dyx)
hotel search "隔音好吗" --strategy all  # 全部跑一遍对比

# 加 LLM
hotel search "性价比怎么样" --llm     # 需 .env 配 DeepSeek API Key
```

默认输出 JSON（ai-friendly），`--no-json` 有人类可读的格式化输出，含贡献者标注。

### 4.3 召回算法综合对比

10 queries, text-based relevance (P@K = Precision@K, 文档与查询关键词匹配则视为相关):

| Metric | BM25 (dyx) | RAG (Yuhao) | Fusion (dyx) | CLIP (Leuca) |
|--------|:---------:|:----------:|:----------:|:----------:|
| P@1 | **100.0** | **100.0** | **100.0** | 50.0 |
| P@5 | **100.0** | 96.0 | **100.0** | 36.0 |
| P@10 | **98.0** | 75.0 | **98.0** | 39.0 |
| MRR | **100.0** | **100.0** | **100.0** | 59.6 |
| avg延迟 | 2.64 ms | ~500 ms* | 5-10 ms | ~500 ms* |

> *RAG 和 CLIP 的 ~500ms 包含首次模型加载（约 2s），后续查询复用缓存。

**排名：**
1. **BM25 (dyx)** — 综合得分 100。简单、快、准，在这个数据集上是最好的选择。
2. **RRF Fusion (dyx)** — 综合得分 100。BM25 + CLIP 融合，在 BM25 够好时无退化。
3. **Agentic RAG (Yuhao)** — 综合得分 98。需要结构化回答时优于 BM25，但慢且需要 LLM。
4. **CLIP Multimodal (Leuca)** — 综合得分 48。图文检索的文本相关性天然偏低，需图片级评测。

### 4.4 与 Web Demo 的关系

两者共享同一套 `_shared/retriever` 和 `_shared/router` 核心逻辑。

## 八、一些反思

### 8.1 做对了的

- **BM25-first 策略**：在实际查询中 BM25 覆盖了几乎所有常见问题。`_run_rag` 里先试 BM25、够了就跳过的逻辑，省掉了不必要的 LLM 调用和延迟。
- **意图分类的中文 10 维体系**：简单、快（10μs）、准（96.4%）。酒店评论查询的意图空间本身就有限——用户翻来覆去问的就是早餐、游泳池、位置、服务这些事。
- **系统化测试体系**：从最初的 25 条单酒店测试扩展到 725 条多酒店测试，每个算法都有标准的召回指标，路由器有单独的精度+延迟评测。

### 8.2 做得不够好的

- **CLIP 多模态检索：** Strict Hit@10 只有 5.4%，LTR 优化后 Hit@5 撞 7% 天花板——问题不在排序在召回。
- **SFT 没部署成服务：** LoRA 权重产出了但没合回去 serve，还是半成品。
- **意图分类器的双重维护：** `dyx/engine.py` 和 `_shared/router.py` 里有两套关键词列表，改了其中一个另一个不同步。
- **RAG 的 P@10 偏低：** 只返回 8 条 evidence，top-rank 密度不如 BM25。

### 8.3 如果再做一遍

1. **先定接口协议再各自实现**——整合阶段花的时间比预计多了一倍。
2. **BM25 够了就别上 CLIP**——5.4% Hit@10 的价值与 CLIP 的复杂度不成正比。
3. **测试要写早**——测试套件是第三阶段才写的，前面全靠手动试。
4. **实验设计要前置**——A/B 测试、路由对比、SFT 推理这些实验，如果一开始就规划好，最后不至于"设计了但没执行"。
