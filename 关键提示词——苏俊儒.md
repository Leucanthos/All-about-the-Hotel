# 关键提示词

苏俊儒

```
你先思考，写出架构。然后写出测试脚本，最后再开发。
```

确立范式。

```
"可以带8个人入住双人房吗"
当前结果：被分到"实务咨询"，展开退房效率、入住时间、前台服务、宠物政策四个维度。
回复里一堆跟"8个人"无关的评论。

这个 case 暴露了一个根本问题：用户查询分成两种——
A 类：要一个明确的事实（能住几人、有没有洗衣机、能抽烟吗）
B 类：要综合分析和建议（带宝宝合适吗、度蜜月体验怎么样）

A 类不应该走维度展开。它不需要 LLM 研判，直接拿原问题去搜评论，有结果就展示，没有就说没找到。
B 类才需要拆维度、多路检索、LLM 综合研判。

按这个分类标准重新设计路由。
```

这条提示词给出了：具体 bad case、错误分析、分类标准、每类的处理方式。LLM 据此重构了整个路由架构——practical 直搜 vs advisory 维度展开两条路径——不需要用户写一行技术方案。

```
"有洗衣机吗"走的是"实务咨询"场景，但仍然展开了四个维度去搜。
现在的问题是：practical 场景所有查询共享同一组核心维度（退房、入住、前台），
但大部分 practical 查询跟这些维度毫无关系。

不要给 practical 预设任何维度。practical 查询直接用原问题的关键词做 BM25+RAG 检索，
6 个事件完成，不走维度展开和 LLM 研判。
```

同一个问题的第二层修复。第一条提示词搭建了两条路径的框架，这条提示词发现 practical 路径内部的维度选择仍然有问题，进一步精简。


```
你这个枚举式的正则匹配能覆盖 80% 的查询，剩下 20% 怎么办？
不能每次都靠人工补正则，应该有一个 LLM 兜底的机制：
关键词先拦截（免费、高频），关键词没命中再走 LLM 分类。
LLM 的 prompt 里要告诉它 practical 场景是什么，给出具体例子，让它能泛化到正则覆盖不了的查询。
```

指出当前方案的覆盖率上限（80%），给出改进方向（两层路由），但不指定 LLM prompt 的具体措辞——那是 LLM 自己的事。

```
错了。关键词优先是对的，但不是因为覆盖面——是因为成本。
关键词匹配不花钱、微秒级，LLM 调用要钱、要等 500ms。
所以关键词是第一道筛子，LLM 是兜底网。
关键词已经能截住 80%，不要因为想推到 100% 而把所有查询都发给 LLM。
```

纠正 LLM 的一个错误倾向（想把所有查询交给 LLM），给出的纠正理由不是主观偏好，而是硬约束（成本、延迟）。带数字的约束比带形容词的约束有效得多。

```
"酒店提供免费避孕套吗"被分到了"情侣蜜月"，查了安静程度、景观私密、餐饮体验。
这个错误太离谱了。反思两个问题：
1. 你的关键词预检为什么没拦截这个查询？
2. 这类"询问具体用品"的查询你还能想到多少种？全部生成出来，写成测试用例。
然后补关键词、修代码、跑通全部新测试。
```

从"离谱吗"升级为：要求 LLM 自己反思为什么没拦住 + 自己生成同类 case + 补关键词 + 加测试 + 全部跑通。一条提示词触发一个完整的修复循环。

```
你的测试覆盖够不够？不是问数量，是问覆盖策略：
- 关键词预检的 14 个类别每个有没有至少 2 条？
- 有没有故意刁难的边缘查询？比如"带大象入住"、"烧香拜佛"、"开 party"
- 有没有确保不该命中的查询不会误命中？比如"带老人去"不应该被 practical 拦截
```

追问测试的覆盖策略而非测试数量。给出了三个具体的覆盖维度，LLM 按这个框架自查并补全。
